## 2. 系统概述 (System Overview)

### 2.1 系统目标

**核心目标:** 为瑜伽馆提供一套完整的会员管理、课程预约、支付结算和客户关系维护的一体化解决方案，提升运营效率和会员体验。

**业务目标:**
1. **提升会员管理效率** - 实现会员信息数字化，支持会员卡、课时包、积分等多种会员体系
2. **优化课程预约流程** - 支持微信小程序预约、取消预约、自动提醒，降低爽约率
3. **提高教练排课效率** - 可视化排课系统，自动冲突检测，减少人工协调成本
4. **完善支付结算体系** - 支持多种支付方式，自动对账，减少财务工作量
5. **增强客户粘性** - 通过积分体系、会员等级、消息推送等功能提升会员活跃度

**技术目标:**
1. **高可用性** - 系统可用性达到 99.9%，支持 7×24 小时服务
2. **易扩展** - 支持业务模块灵活扩展，可快速迭代新功能
3. **数据安全** - 保障会员隐私数据、支付数据安全
4. **多端适配** - 支持微信小程序、管理后台、数据看板等多端访问
5. **性能要求** - 核心接口响应时间 < 200ms，支持 1000+ 并发用户

### 2.2 用户群体

| 用户类型 | 描述 | 核心需求 |
|----------|------|----------|
| **普通会员** | 瑜伽馆的注册会员 | 微信小程序登录、查看课程表并预约、查看个人会员卡/课时、在线支付、查看积分 |
| **VIP 会员** | 购买年卡/私教包的高价值会员 | 专属课程预约通道、会员等级特权展示、积分兑换礼品、生日福利提醒 |

| **前台/客服** | 瑜伽馆前台接待和客服人员 | 会员信息管理、现场预约和签到、会员卡办理、问题处理记录 |
| **运营管理员** | 负责课程、活动、消息推送的运营人员 | 课程发布和管理、活动配置和推送、数据统计分析、会员标签管理 |
| **系统管理员** | 系统运维和配置管理人员 | 用户权限管理、系统配置、日志查看、数据备份恢复 |
| **财务人员** | 负责财务对账和结算 | 收入统计、对账报表、退款审核、发票管理 |

## 3. 系统架构 (System Architecture)

### 3.1 架构风格

**架构模式:** 模块化单体架构 (Modular Monolith) + 事件驱动

**选择理由:**
1. **业务复杂度适中** - 瑜伽馆系统业务边界清晰，初期无需过度拆分微服务
2. **开发效率高** - 单体架构减少分布式事务、服务间调用等复杂度
3. **便于迭代** - 模块化设计支持后续按需拆分为微服务
4. **运维成本低** - 减少服务数量，降低运维和监控成本
5. **数据一致性好** - 本地事务保证数据一致性，避免分布式事务复杂性

**架构演进路线:**
- 阶段一 (当前): 模块化单体架构，快速迭代验证业务
- 阶段二 (用户量增长): 核心服务拆分（用户服务、订单服务独立）
- 阶段三 (规模化): 微服务架构，按业务域拆分

## 4. 模块设计 (Module Design)

### 4.1 模块划分

系统按业务功能域划分为以下核心模块：

```

├── 用户中心模块
│   ├── 用户认证
│   ├── 会员管理
│   └── 等级与积分
├── 课程管理模块
│   ├── 课程发布
│   ├── 课程表管理
│   └── 教练管理
├── 预约服务模块
│   ├── 预约下单
│   ├── 预约管理

├── 支付中心模块
│   ├── 订单管理

    ├── 业务报表

    └── 运营看板
```

#### 4.1.1 用户中心模块

**职责:** 负责用户身份认证、会员信息管理、会员等级与积分体系管理

**功能列表:**
- 微信登录授权、手机号绑定
- 会员信息 CRUD（姓名、性别、生日、体态特征等）
- 会员卡管理（次卡、月卡、季卡、年卡）
- 课时包管理
- 会员等级体系（普通会员、银卡、金卡、钻石卡）
- 积分获取与消耗
- 优惠券管理

**关键组件:**
- `AuthService` - 认证服务，处理登录授权逻辑
- `MemberService` - 会员服务，管理会员信息
- `MembershipCardService` - 会员卡服务
- `PointsService` - 积分服务，积分计算和流水
- `LevelService` - 等级服务，等级升降级逻辑

#### 4.1.2 课程管理模块

**职责:** 负责课程模板管理、课程表编排、教练排课

**功能列表:**
- 课程模板管理（课程名称、描述、难度、时长、适用人群）
- 课程表管理（排课、调课、取消）
- 教练管理（教练信息、可授课程、可排课时间）
- 教室管理（教室容量、设备信息）
- 课程库存管理（人数限制、预约限制）

**关键组件:**
- `CourseTemplateService` - 课程模板服务
- `ScheduleService` - 排课服务
- `CoachService` - 教练服务
- `ClassroomService` - 教室服务
- `InventoryService` - 库存服务

#### 4.1.3 预约服务模块

**职责:** 负责课程预约、取消、签到核销等预约全生命周期管理

**功能列表:**
- 课程预约（选择课程、使用会员卡/课时）
- 预约取消（支持自动退款）
- 预约记录查询
- 扫码签到/核销
- 预约限制（黑名单、预约次数限制）
- 预约提醒（课前提醒）

**关键组件:**
- `BookingService` - 预约服务，核心业务逻辑
- `BookingRuleService` - 预约规则服务
- `CheckinService` - 签到服务
- `ReminderService` - 提醒服务

**核心流程 - 预约流程:**
```mermaid
sequenceDiagram
    participant 用户 as 用户
    participant 小程序 as 小程序
    participant API as API 网关
    participant 预约 as 预约服务

    用户中心-->>预约：卡片有效
    预约->>预约：创建预约记录
```

## 7. 安全设计 (Security Design)

### 7.1 认证与授权

**认证方式:**
- 微信小程序登录（基于 OAuth 2.0）
- JWT Token 认证（有效期 2 小时）
- 手机号 + 验证码登录（备用方案）

**授权模型:** RBAC（基于角色的访问控制）

| 角色 | 权限范围 |
|------|----------|
| 普通会员 | 查看自己的信息、预约课程、购买会员卡 |
| VIP 会员 | 普通会员权限 + 专属课程预约、积分兑换 |
| 教练 | 查看课程表、学员签到、学员信息 |
| 前台 | 会员管理、预约管理、现场签到 |
| 运营管理员 | 课程管理、活动配置、数据统计 |
| 系统管理员 | 用户管理、权限配置、系统设置 |
| 财务人员 | 订单管理、退款审核、财务报表 |

**会话管理:**
- Token 有效期：2 小时
- Refresh Token 有效期：7 天
- 支持 Token 黑名单（注销、改密时使旧 Token 失效）

### 7.2 数据安全

**传输加密:**
- 全站 HTTPS（TLS 1.3）
- 敏感接口（支付、密码）使用国密 SM2 加密

**存储加密:**
- 密码：bcrypt 哈希（cost=12）
- 手机号：AES-256 加密存储
- 敏感字段（身份证、银行卡）：加密存储

**密钥管理:**
- 使用云服务密钥管理系统（KMS）
- 定期轮换密钥
- 密钥分级管理（开发、测试、生产分离）

### 7.3 安全防护

| 风险类型 | 防护措施 |
|----------|----------|
| XSS 攻击 | 输入验证、输出转义、CSP 策略 |
| CSRF 攻击 | Token 验证、SameSite Cookie |
| SQL 注入 | 参数化查询、ORM 框架 |
| 接口滥用 | 限流（令牌桶）、防刷策略 |
| 数据泄露 | 脱敏展示、访问审计 |

### 7.4 审计与日志

**审计内容:**
- 用户登录/登出
- 敏感操作（支付、退款、权限变更）
- 数据批量导出
- 异常访问行为

**日志级别:**
- ERROR: 系统错误、异常
- WARNING: 警告信息（如限流触发）
- INFO: 业务操作日志
- DEBUG: 调试信息（仅开发环境）

**日志存储:**
- 集中式日志系统（ELK Stack）
- 日志保留 180 天
- 敏感信息脱敏

## 8. 非功能性需求 (Non-Functional Requirements)

### 8.1 性能需求

| 指标 | 目标值 | 测量方法 |
|------|--------|----------|
| 响应时间 (P95) | < 200ms | 压测工具 (wrk/JMeter) |
| 吞吐量 | > 1000 QPS | 压测工具 |
| 并发用户数 | > 1000 | 压测工具 |
| 页面加载时间 | < 2s | Lighthouse |
| 接口可用性 | > 99.9% | 监控系统 |

### 8.2 可用性

- **SLA:** 99.9%（全年停机时间 < 8.76 小时）
- **容灾方案:** 数据库主从复制，应用多实例部署
- **故障恢复:** RTO < 30 分钟，RPO < 5 分钟
- **备份策略:** 数据库每日全量备份 + 每小时增量备份

### 8.3 可扩展性

- **水平扩展:** 应用服务无状态，支持动态扩容
- **垂直扩展:** 数据库读写分离，支持分库分表
- **模块化设计:** 业务模块可独立部署和扩展

### 8.4 可维护性

- **代码规范:** 遵循 ESLint/Prettier 规范
- **文档要求:** API 文档自动生成（Swagger），关键代码注释
- **监控告警:** Prometheus + Grafana，多渠道告警（邮件/短信/钉钉）
- **日志规范:** 统一日志格式，支持链路追踪

## 9. 部署与运维 (Deployment & Operations)

### 9.1 部署方案

**环境划分:**
| 环境 | 用途 | 配置 |
|------|------|------|

**部署方式:**
- 开发/测试环境：Docker Compose
- 生产环境：Kubernetes（后期）
- 发布策略：滚动更新（零停机）
- 回滚策略：一键回滚到上个稳定版本

### 9.2 监控方案

**系统监控:**
- CPU、内存、磁盘、网络
- 容器健康状态

**应用监控:**
- 接口响应时间、错误率、QPS
- JVM/Node.js 运行时指标
- 数据库连接池、慢查询

**业务监控:**
- 每日新增会员数
- 课程预约率、到场率
- 订单金额、支付成功率

**告警渠道:** 邮件、短信、钉钉、企业微信

### 9.3 备份策略

**数据库备份:**
- 全量备份：每日凌晨 2 点
- 增量备份：每小时
- 备份存储：异地备份（不同可用区）
- 恢复演练：每季度一次

**代码备份:**
- Git 仓库多地域备份
- 重要配置文件加密备份
